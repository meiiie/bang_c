## Root causes (verified)

### A. Confidence is fake (teammate point #3 — CONFIRMED, highest priority)
`solver.py` returns confidence as **hard-coded constants per code path**, not a
calibrated probability:
- `0.88` if solver==verifier else `0.72` (`solver.py:440`)
- `0.86/0.76`, `0.82`, `0.80/0.70`, `0.74`, `0.68`, tournament `0.72 / 0.78+…` …

Because the verifier is the same model agreeing cheaply (see C), 79.5% land at 0.88.
The risk reviewer (`review.py`, threshold 0.84) therefore flags ~2% while ~14% are
wrong. **Consequence:** trace review, review-tasks, and "auto" compute-triage are all
driven by a number that carries almost no information.

### B. Diacritic-collision routing (teammate point #1 — CONFIRMED, not stale)
`text.normalize_text()` strips Vietnamese diacritics (NFD + drop `Mn` + đ→d) before
keyword matching in `classifier.py`. Stripping **merges semantically distinct words**:
`tỉnh` (province) / `tính` (calculate|nature) / `tinh` (essence) → all `"tinh"`.

The calc marker `tính`→`tinh` then matches non-math text. There ARE guards
(`_BROAD_CALCULATION_MARKERS={tinh,gia tri,bao nhieu}` require
`_has_quantitative_signal`), **but the guard only checks for any digit/math char in
question+choices** (`classifier.py:157`) — far too loose: Vietnamese civics/history
items routinely contain stray numbers (years, "Điều 5", "5 tốt", percentages).

Verified in v5 trace — all four teammate examples **still misroute to calculation**:
`test_0022 (tỉnh Cà Mau)`, `test_0080 (Tính nhân văn…)`, `test_0153 (Tính khách
quan…)`, `test_0190 (…giá trị tư tưởng)` → `kind=calculation, variant=calculation`.
The `calculation` bucket (74/463 = 16%) is **inflated by false positives**, sending
non-math questions through the calculation prompt + bespoke formula adjudicators (see
E). This is whack-a-mole: every fix adds another guard.

> Diagnosis of the *root*, not the symptom: the problem is not "substring vs token
> boundary" (the code already uses word boundaries, `_contains_phrase`,
> `classifier.py:147`). The problem is **stripping diacritics at all** — Vietnamese
> tone marks are meaning-bearing, so the collision happens at the whole-word level.

### C. Verifier is not independent (teammate point #4 — CONFIRMED)
`_verify()` (`solver.py:743`) calls the **same model, same context, answer-only**
(`build_verifier_prompt`, max_tokens 64). Same-model self-agreement confirms the
solver's own bias → inflates the 0.88 bucket. The contest allows **two model families
(Gemma-4 AND Qwen3.5)** — a different model as adjudicator is the obvious upgrade.

### D. Reasoning is suppressed (NEW finding — likely a large accuracy leak)
`SYSTEM_PROMPT` (`prompting.py:10`): *"Return exactly one uppercase option letter. Do
not include explanation"* and bundles use `max_tokens≈96`. The model is **forbidden
from chain-of-thought**. For calculation and multi-step reasoning items, letter-only
output is known to drop accuracy sharply. The harness then tries to compensate with
hand-written Python math (E) — treating a symptom of the prompt design.

### E. Overfit adjudicators (teammate context — CONFIRMED, biggest *transfer* risk)
- `calculation.py`: ~7 bespoke solvers (`gdp_inflation`, `cylinder_fill_rate`,
  `buffer_ph`, `carbon_percent_from_co2`, `wire_length_power_current_resistivity`,
  `ideal_transformer_primary_current`, `quadratic_production_max_labor`) + decision
  rules. These encode **specific public-test questions**. They cannot generalize to
  2000 unseen, multilingual items and can mis-fire on look-alikes.
- `principles.py`: ~6 hard-coded VN domain rules (preschool accreditation, anti-state
  refusal, three-phase wattmeter, Hồ Chí Minh communism adaptation, …). Same problem.
- `SYSTEM_PROMPT` / verifier prompt hard-code **"Vietnamese"**; `_split_context_and_query`
  only knows VI/EN/ES markers. **Zero coverage for KO/ZH/JP** and other private-test
  languages.

### F. Single-label routing + god functions (points #2, #6 — PARTLY / CONFIRMED)
- Features ARE computed multi-label (`classifier.py:52-56`) but **collapsed to one
  `kind`** via if/elif (`:79-98`). Teammate's specific ordering claim is **WRONG**:
  calculation is checked *before* many_choice (`:82` vs `:85`), not after.
- God functions confirmed: `solver.py`=808 lines, `_solve_direct`=384, `run.py`=648.
  `_solve_direct` rebuilds the `Prediction(...)` block ~6× with duplicated
  adjudicator cascades — a refactor magnet.
- Tournament tie (point #5): real code smell (A-biased `tied_answers[0]`,
  `solver.py:583`) but **0 ties occurred in v5** → low real-world impact today.

## What this means for the 2000-question multilingual private test

The public 85.53 is propped up by **Vietnamese-specific heuristics and 463-specific
adjudicators**. On a diverse multilingual set those props don't just stop helping —
several actively **hurt** (false calc routing, wrong-language prompts, mis-firing
formula solvers). The score could drop *and* we wouldn't see it (fake confidence).

**The user's instinct is correct on both counts:**
1. *Why strip Vietnamese diacritics?* Answer: only the legacy keyword router needs it;
   the model never sees stripped text and understands diacritics natively. Stripping
   buys brittle keyword matching at the cost of word collisions. **Keep diacritics;
   reduce reliance on keyword routing.**
2. *Don't overfit to 463.* The bespoke formula/principle rules + VI-only prompts are
   exactly that. Replace with general, language-agnostic, model-driven reasoning.
